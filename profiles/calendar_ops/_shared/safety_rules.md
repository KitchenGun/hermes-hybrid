# Safety Rules — calendar_ops

쓰기(create/update/delete) 잡에 공통으로 적용되는 안전 규칙.
잡 YAML의 `prompt` 필드는 이 규칙을 자동 상속한다.

## 1. 확인 게이트 (Confirmation Gate)

모든 쓰기 작업은 **실행 전 사용자 확인**을 거친다.

```
사용자 요청
    ↓
LLM: 의도 파싱 (intent_schema.json 따름)
    ↓
[확인 메시지 전송] ← 여기서 정지
    ↓ 사용자 "확인" 수신 시에만
실제 tool 호출 (google_calendar.create_event 등)
```

확인 메시지에 반드시 포함:
- 동작명 (추가/수정/삭제)
- 대상 이벤트의 모든 필드 (제목·시간·장소·참가자)
- 수정 시: 변경 전 vs 변경 후 대조
- 삭제 시: 이벤트 ID와 원본 제목

## 2. 한 번에 하나의 이벤트

- 배치 작업 금지: "내일 일정 다 지워줘" 같은 요청은 **개별 확인**으로 분해
- 반복 이벤트 수정 시: "이 이벤트만 / 앞으로의 이벤트 / 모든 반복" 중 명시 선택

## 3. Idempotency 보장

- **create**: 같은 (title, start, end, calendar_id)가 이미 있으면 중복 생성 금지, 기존 이벤트 ID 반환
- **update**: event_id로만 특정. 제목·시간 기반 매칭 금지
- **delete**: event_id로만 특정. "방금 그거" 같은 대명사 참조 시 재질문

## 4. 재시도 금지

쓰기 작업은 `max_retries: 0`. 실패 시:
1. Ledger에 실패 기록
2. 사용자에게 원인 즉시 보고
3. **절대 자동 재실행하지 말 것** (중복 생성·이중 삭제 위험)

## 5. Rollback 기록

삭제 작업 시 원본 이벤트 전체 스냅샷을 Ledger에 기록:
```json
{
  "event_type": "calendar_delete",
  "event_id": "abc123",
  "snapshot": {
    "title": "...",
    "start": "...",
    "end": "...",
    "attendees": [...],
    "description": "..."
  },
  "deleted_at": "2026-04-20T14:30:00+09:00"
}
```

Discord 명령 `!undo calendar <event_id>` 로 복원 가능.

## 6. OAuth 토큰 처리

- 만료 감지: 401/403 응답 수신 시 즉시 중단, 재시도 금지
- 재인증 요청: 사용자에게 "재인증 필요" 명시, 토큰 파일(`auth.json`) 백업 후 재발급 유도
- 토큰 문자열은 **로그·Ledger에 절대 기록 금지**. `security.secret_redaction: true` 신뢰.

## 7. 모호한 참조 해소

다음 표현은 **항상 재질문**:
- "방금 그거", "아까 말한 거"
- "다음 미팅" (여러 개 가능)
- "회의" (제목이 같은 여러 이벤트)

재질문 포맷:
```
어떤 일정을 말씀하시는지 명확히 해주세요:
  1) 4월 21일 14:00 — 디자인 리뷰
  2) 4월 22일 10:00 — 주간 미팅
```

## 8. 개인정보 보호

- 참가자 이메일: 조회 결과 출력 시 도메인만 표시 (`user@example.com` → `@example.com`) 옵션 가능
- 비공개 이벤트(`visibility: private`): 제목을 "🔒 비공개"로 마스킹, 세부 정보 노출 금지
- 타인 초대 이벤트: 주최자 대리 수정 금지

## 9. 충돌 감지

일정 추가 시 기존 일정과 시간 겹침 감지:
- **경고만 제시**, 자동 수정 금지
- 사용자 선택지: "그대로 추가 / 시간 변경 / 취소"

## 10. 긴급 정지

다음 조건에서 잡 전체 중단:
- OAuth 토큰 무효
- Google API 5xx 응답 3회 연속
- Ledger 기록 실패 (쓰기 잡에만 해당 — 감사 누락 방지)
